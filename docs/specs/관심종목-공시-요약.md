# 스펙: 관심종목 공시 요약 웹앱

> 상태: ready-for-agent (이슈 트래커 미설정 — 저장소 파일로 보관)
> 관련 문서: [CONTEXT.md](../../CONTEXT.md) · ADR [0001](../adr/0001-mock-first-dart-only-source.md) [0002](../adr/0002-no-auth-device-local-watchlist.md) [0003](../adr/0003-ai-summary-strict-factual.md) [0004](../adr/0004-disclosure-level-cache-hybrid-generation.md)

## Problem Statement

개인투자자는 자신이 관심 있게 지켜보는 여러 종목에 대해 새로 올라오는 공시를 놓치지 않고 빠르게 파악하고 싶다. 그러나 공시는 종목마다 흩어져 있고, 원문은 길고 정형화된 법정 문서라 핵심을 한눈에 파악하기 어렵다. "내 관심종목에 최근 무슨 일이 있었는지"를 훑는 데 시간과 노력이 많이 든다.

## Solution

사용자가 관심종목을 등록해 두면, 종목별 접힌 카드 목록(기본)으로 한눈에 전체 종목을 훑고, 카드를 클릭해 해당 종목의 최근 공시를 인라인으로 펼쳐 보며, 개별 공시를 클릭하면 **구조화 발췌(정형 항목) + AI가 생성한 사실 중립 요약 3줄**을 모달로 확인하는 웹앱. 사용자는 기간(오늘/1주/1개월)과 보기 방식(종목별/최신순 피드)을 전환할 수 있고, 아직 확인하지 않은 새 공시는 NEW 뱃지로 구분된다. 로그인 없이 즉시 사용할 수 있으며, 관심종목과 읽음 상태는 사용자의 브라우저에 저장된다.

## User Stories

1. 개인투자자로서, 나는 관심종목을 종목명 또는 종목코드로 검색해 등록하고 싶다, 그래야 추적할 종목을 내가 직접 구성할 수 있다.
2. 개인투자자로서, 나는 등록한 관심종목을 목록(칩)으로 보고 개별 삭제하고 싶다, 그래야 더 이상 보지 않을 종목을 정리할 수 있다.
3. 개인투자자로서, 나는 로그인 없이 바로 서비스를 쓰고 싶다, 그래야 가입 절차의 마찰 없이 즉시 이용할 수 있다.
4. 개인투자자로서, 나는 다시 방문했을 때 이전에 등록한 관심종목이 그대로 남아 있길 바란다, 그래야 매번 다시 등록하지 않아도 된다.
5. 개인투자자로서, 나는 내 관심종목들의 공시를 하나의 시간순 피드로 모아 보고 싶다, 그래야 전체적으로 최근 무슨 일이 있었는지 훑을 수 있다.
6. 개인투자자로서, 나는 피드를 "종목별 보기"로 전환하고 싶다, 그래야 특정 종목에 어떤 공시가 있었는지 집중해서 볼 수 있다.
7. 개인투자자로서, 나는 조회 기간을 오늘/1주/1개월로 전환하고 싶다, 그래야 원하는 시간 범위의 공시만 볼 수 있다.
8. 개인투자자로서, 나는 각 공시의 핵심을 3줄 요약으로 보고 싶다, 그래야 원문을 다 읽지 않고도 내용을 파악할 수 있다.
9. 개인투자자로서, 나는 공시의 정형 항목(공시유형·금액·일자·상대방 등)을 발췌로 보고 싶다, 그래야 숫자와 사실을 빠르게 확인할 수 있다.
10. 개인투자자로서, 나는 요약이 전망·평가·매매의견 없이 사실만 담고 있길 바란다, 그래야 객관적 정보로만 판단할 수 있다.
11. 개인투자자로서, 나는 각 공시 요약에서 원문(DART 등)으로 바로 이동하고 싶다, 그래야 필요 시 원문을 직접 확인할 수 있다.
12. 개인투자자로서, 나는 각 요약에 "투자권유·자문이 아니다"라는 면책 문구가 명시되길 바란다, 그래야 정보의 성격을 오해하지 않는다.
13. 개인투자자로서, 나는 아직 확인하지 않은 새 공시가 NEW 뱃지로 표시되길 바란다, 그래야 새로운 소식을 한눈에 구분할 수 있다.
14. 개인투자자로서, 나는 공시 카드를 열람하면 그 공시가 읽음 처리되길 바란다, 그래야 이미 본 것과 안 본 것이 구분된다.
15. 개인투자자로서, 나는 관심종목을 하나도 등록하지 않았을 때 무엇을 해야 할지 안내받고 싶다, 그래야 처음에도 막히지 않는다.
16. 개인투자자로서, 나는 각 공시 카드에서 어느 종목의 공시인지(종목명·코드) 뱃지로 확인하고 싶다, 그래야 피드에서 종목을 혼동하지 않는다.
17. 개인투자자로서, 나는 공시가 얼마나 최근에 올라왔는지(예: "3시간 전") 알고 싶다, 그래야 최신성을 가늠할 수 있다.
18. 개인투자자로서, 나는 주요 종목(KOSPI200 등)의 공시 요약을 지연 없이 즉시 보고 싶다, 그래야 대기 시간 없이 확인할 수 있다.
19. 개인투자자로서, 나는 유니버스 밖 종목의 공시도 (약간의 생성 대기 후) 요약으로 볼 수 있길 바란다, 그래야 어떤 종목을 담아도 동일한 경험을 얻는다.
20. 개인투자자로서, 나는 모바일 브라우저에서도 화면이 편하게 보이길 바란다, 그래야 이동 중에도 확인할 수 있다.

## Implementation Decisions

- **플랫폼/스택**: Next.js(App Router) + TypeScript 풀스택 단일 저장소. mock 단계에서는 화면과 상태를 클라이언트에서 구성하고, 실데이터 단계에서 서버(Route Handler)·배치 크론을 추가한다.
- **"공시 요약"의 구성 (A+B 하이브리드)**: 각 요약은 두 층위를 가진다 — (1) 구조화 발췌(정형 항목 배열), (2) AI 요약(사실 중립 서술 3줄). CONTEXT.md의 `공시 요약` 정의를 따른다.
- **데이터 출처 (ADR-0001)**: 외부 출처는 DART 단일. 공시 원문과 종목 마스터(기업개황 corpCode)를 DART에서 얻는다. 개발은 더미 데이터 우선, 이후 동일 인터페이스 뒤에서 DART로 교체.
- **주요 종목 유니버스 (ADR-0001, 0004)**: KOSPI200 고정 리스트로 정의(정기 리밸런싱 시 수동 갱신). 사용자 데이터와 무관.
- **종목 등록 범위 (US-19)**: 등록 가능한 종목은 유니버스에 제한되지 않는다. 검색·corp_code 해석은 DART 기업개황에서 상장사만 추린 **전체 색인**(`lib/dart/corp-index.json`, 서버 전용)으로 처리하고, 클라이언트는 서버 라우트 `/api/stocks`(검색)·`/api/disclosures`(공시)만 통해 접근한다. 유니버스 20종목은 배치 사전요약 대상·"유니버스" 배지·시장 라벨 오버레이로만 남긴다. 관심종목은 이름 표시를 위해 localStorage에 `{code, name}`로 저장(구버전 `string[]`은 로드 시 마이그레이션).
- **인증·저장 (ADR-0002)**: 로그인·계정 없음. 관심종목과 읽음 상태는 브라우저 localStorage에만 저장. 서버는 개별 사용자를 식별하지 않는다.
- **요약 생성·캐시 (ADR-0004)**: 요약은 사용자 무관하게 **공시 단위**로 생성·캐시되어 모든 사용자가 공유한다. 생성은 하이브리드 — 유니버스 공시는 스케줄러가 사전 배치 요약, 유니버스 밖은 조회 시 on-demand.
- **AI 요약의 경계 (ADR-0003)**: 전망·평가·매매의견 금지, 공시에 적힌 사실만 중립 서술. 요약마다 면책 문구 + 원문 링크 노출. LLM은 Claude(대량 공시 Haiku, 중요 공시 상위 모델로 이원화, 모델 ID·단가는 착수 시 확정).
- **조회 기간**: `today` / `week`(기본) / `month` 토글. 각 종목·기간 조합으로 공시를 필터링.
- **화면 구성(3단계 드릴다운)**: 종목별 보기가 기본이며, 접힌 종목 카드 목록으로 시작한다(레벨0 — 종목명·코드·건수·미열람 배지만 노출, 한눈에 종목 목록 파악). 카드를 클릭하면 해당 종목의 공시가 카드 안에 인라인으로 펼쳐진다(레벨1 — 아코디언, 여러 카드 동시에 펼침 가능, 재방문 시 기본은 전부 접힘). "최신순" 토글은 시간순 통합 피드로 평면 나열(US-5 유지, 행 클릭 시 바로 모달). 필터(공시유형)는 없음.
- **상세·원문 링크(레벨2)**: 개별 공시 행을 클릭하면 모달(다이얼로그)이 열리고, 그 안에 구조화 발췌·AI 요약·면책 문구·원문 링크가 함께 표시된다. 모달은 Esc·배경 클릭·닫기 버튼으로 닫히고, 포커스는 트리거로 되돌아간다.
- **새 공시 표시**: 읽지 않은 공시에 NEW 뱃지(종목 카드 헤더에는 종목 단위 미열람 건수 배지). 모달 열람 시 읽음 처리하고 localStorage에 읽음 ID 집합을 저장.
- **정정공시**: mock 단계에서는 단순 나열(A). 실데이터 단계에서 정정 우선 + 원본 접기(B)로 확장 예정.
- **데이터 접근 seam(경계)**: 화면은 공시·종목을 직접 mock 모듈에서 읽지 않고, 하나의 데이터 제공자 인터페이스(예: `getStockMaster()`, `getDisclosures(codes, period)`)를 통해서만 접근한다. mock 구현과 실 DART 구현이 이 단일 인터페이스를 각각 만족한다. **이 seam이 mock↔실데이터 교체 지점이자 주요 테스트 지점이다.**

## Testing Decisions

- **좋은 테스트의 기준**: 구현 세부가 아니라 외부로 관찰 가능한 행위를 검증한다. 컴포넌트 내부 상태나 함수 시그니처가 아니라 "사용자에게 무엇이 보이고, 어떤 상호작용에 어떻게 반응하는가"를 확인한다.
- **최상위 단일 seam에서 테스트**: 데이터 제공자 인터페이스(`getStockMaster`, `getDisclosures`)를 seam으로 삼아, 화면 로직(기간 필터링, 피드 정렬, 종목별 그룹핑, NEW 판정, 요약 카드 렌더)을 이 경계 위에서 테스트한다. mock 제공자를 주입해 결정적(deterministic) 입력으로 검증한다.
- **테스트 대상 행위**:
  - 관심종목 등록/삭제가 목록과 피드에 반영되는가.
  - 기간 토글이 노출 공시 집합을 올바르게 바꾸는가.
  - 피드↔종목별 보기 전환이 동일 데이터를 올바르게 재구성하는가.
  - 미열람 공시에 NEW가 뜨고, 열람 후 사라지며, 그 상태가 재방문 시 유지되는가(localStorage).
  - 요약 카드에 구조화 발췌·3줄 요약·면책 문구·원문 링크가 모두 렌더되는가.
- **결정성 확보**: mock 공시의 시각은 "현재 시각 기준 N시간 전"으로 주입해, 테스트에서 현재 시각을 고정하면 기간 필터 결과가 결정적이 되도록 한다.
- **Prior art**: 저장소에 아직 테스트가 없음 — 이 스펙의 테스트가 첫 도입 사례가 된다.

## Out of Scope

- 삼성증권 계정 연동(SSO) 및 계정 기반 관심종목 동기화.
- 로그인·회원가입·서버측 개인정보 저장.
- 기기 간 관심종목/읽음 상태 동기화.
- 브라우저 푸시 알림.
- 공시유형 필터 및 "중요도" 판정·뱃지(투자판단 오인 리스크로 보류).
- 실시간 시세·차트·보유종목(포트폴리오) 기능.
- 실데이터 단계의 정정공시 묶기(B) — 후속 스펙에서 다룸.
- 배치 스케줄러/워커 인프라의 실제 구현 — mock 단계에서는 미포함.
- LLM 실연동 — mock 단계에서는 더미 요약 텍스트 사용.

## Further Notes

- KOSPI200은 정기 리밸런싱이 있어 고정 유니버스 리스트는 주기적 수동 갱신이 필요하다.
- AI 요약은 규제상 사실 중립 서술로 엄격히 제한된다(ADR-0003). 프롬프트 설계·검수 시 이 경계를 최우선으로 지킨다.
- 로그인이 없어 "인기 종목"을 사용자 데이터로 산출할 수 없으므로, 배치 대상은 시장 기준 유니버스로만 정의한다(ADR-0002, 0004).
- 이 스펙은 mock 단계 화면을 대상으로 하며, 실데이터(DART) 연동·배치·정정공시·중요도는 후속 스펙으로 분리한다.
